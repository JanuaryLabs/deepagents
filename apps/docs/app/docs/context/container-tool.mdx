`createContainerTool` combines `createDockerSandbox` and `createBashTool` into a single call, giving you a bash tool ready for AI SDK agents running inside an isolated Docker container.

## Basic Usage

```typescript
import { groq } from '@ai-sdk/groq';
import { printer } from '@deepagents/agent';
import {
  ContextEngine,
  InMemoryContextStore,
  agent,
  createContainerTool,
  role,
  skills,
  user,
} from '@deepagents/context';

const sandbox = await createContainerTool({
  packages: ['curl', 'jq'],
  skills: [
    { host: './skills', sandbox: '/workspace/skills' },
  ],
});

try {
  const context = new ContextEngine({
    chatId: 'demo',
    userId: 'user-1',
    store: new InMemoryContextStore(),
  }).set(
    role('You are a system admin with bash access.'),
    skills(sandbox),
    user('Use the most relevant skill to answer.'),
  );

  const assistant = agent({
    name: 'Admin',
    sandbox,
    model: groq('gpt-oss-20b'),
    context,
  });

  await printer.stdout(await assistant.stream({}));
} finally {
  await sandbox.sandbox.dispose();
}
```

## Return Value

`createContainerTool` returns a `ContainerToolResult` (compatible with `AgentSandbox`):

```typescript
const sandbox = await createContainerTool({ ... });
```

| Property | Type | Purpose |
|----------|------|---------|
| `sandbox.bash` | `BashExecuteTool` | Single bash tool for direct use |
| `sandbox.tools` | `{ bash, readFile, writeFile }` | Full toolkit — auto-registered when passed to `agent()` |
| `sandbox.sandbox` | `DockerSandbox` | Container handle — call `sandbox.sandbox.dispose()` for cleanup |
| `sandbox.skills` | `SkillPathMapping[]` | Discovered skills (empty if `skills` option omitted) — pass to `skills(sandbox)` fragment |

Pass the entire result as the `sandbox` field to `agent()`. The agent merges `sandbox.tools` into its toolset automatically.

### The `skills` option

Declare skill directories via `skills: [{ host, sandbox }]`. At container startup the factory:

1. Recursively walks each `host` directory (skipping symlinks and dotfiles).
2. Writes every file under the matching `sandbox` path inside the container.
3. Parses each subdirectory's `SKILL.md` frontmatter into `sandbox.skills`.

This replaces the earlier pattern of manually mounting a host directory and separately configuring `skills({ paths })`. The sandbox is now the single source of truth — you cannot end up with a `skills()` entry that points to a path the LLM can't actually read.

<Callout type="warn">
  The `sandbox` destination of a skill entry must be writable — it's written via
  `writeFiles` on container start. Do not place it under a `readOnly: true` mount,
  or put it in a separate, writable location from your read-only host mounts.
</Callout>

## Strategy Options

All three Docker sandbox strategies are supported:

<Tabs groupId="container-strategy" items={['Runtime', 'Dockerfile', 'Compose']}>
<Tab value="Runtime">

```typescript
const { bash, sandbox } = await createContainerTool({
  image: 'alpine:latest',     // default
  packages: ['python3', 'nodejs'],
  env: {
    NODE_ENV: 'production',
    API_KEY: process.env.API_KEY || '',
  },
  binaries: [
    {
      name: 'presenterm',
      url: {
        x86_64: 'https://github.com/.../presenterm-x86_64-musl.tar.gz',
        aarch64: 'https://github.com/.../presenterm-aarch64-musl.tar.gz',
      },
    },
  ],
  mounts: [
    {
      hostPath: process.cwd(),
      containerPath: '/workspace',
    },
  ],
  resources: { memory: '1g', cpus: 2 },
});
```

</Tab>
<Tab value="Dockerfile">

```typescript
const { bash, sandbox } = await createContainerTool({
  dockerfile: `
    FROM python:3.11-slim
    RUN pip install pandas numpy
  `,
  context: '.',
  mounts: [
    {
      hostPath: process.cwd(),
      containerPath: '/workspace',
    },
  ],
});
```

</Tab>

const { bash, sandbox } = await createContainerTool({
  compose: './docker-compose.yml',

## Bash Tool Hooks

Intercept command execution for logging, auditing, or command transformation:

```typescript
import chalk from 'chalk';

const { bash, sandbox } = await createContainerTool({
  packages: ['curl', 'jq'],
  onBeforeBashCall: ({ command }) => {
    console.log(chalk.blue(`[Container] Running: ${command}`));
    return { command };
  },
  onAfterBashCall: ({ command, result }) => {
    if (result.exitCode !== 0) {
      console.log(chalk.yellow(`[Container] Failed (${result.exitCode}): ${command}`));
    }
    return { result };
  },
});
```

`onBeforeBashCall` can modify the command before execution. `onAfterBashCall` can modify the result before it's returned to the agent.

## With AI SDK Directly

Instead of using the `agent()` wrapper, pass `tools` directly to `generateText` or `streamText`:

```typescript
import { generateText } from 'ai';
import { groq } from '@ai-sdk/groq';
import { createContainerTool } from '@deepagents/context';

const { tools, sandbox } = await createContainerTool({
  packages: ['python3'],
  mounts: [
    {
      hostPath: process.cwd(),
      containerPath: '/project',
      readOnly: false,
    },
  ],
});

try {
  const result = await generateText({
    model: groq('gpt-oss-20b'),
    tools,
    prompt: 'Run python3 --version and tell me what version is installed',
  });

  console.log(result.text);
} finally {
  await sandbox.dispose();
}
```
