title: MinIO / S3 Cloud Storage Volumes
description: Persist sandbox files to MinIO (or any S3-compatible store) with an rclone-backed Docker volume — no library coupling
---

This recipe persists sandbox files to **MinIO** — or any S3-compatible store (AWS S3, Wasabi, Cloudflare R2) — through an [rclone](https://rclone.org/docker/) Docker volume plugin. MinIO is just rclone's `s3` backend with `provider = Minio`, so the `@deepagents/context` side uses the standard `type: 'volume'` primitive: the library never learns about MinIO, rclone, or credentials.

It is the S3 counterpart of the [GCS Cloud Storage recipe](/docs/context/recipes/gcs-cloud-storage) — same plugin, same volume primitive, a different `rclone.conf` stanza. The walkthrough targets **local development on Docker Desktop** (the common case: a MinIO container you spin up yourself), then notes the changes for a real/remote MinIO endpoint.

## Topology (local dev on Docker Desktop)

On Docker Desktop the daemon — and therefore the rclone plugin, the MinIO container, and the sandbox — all live inside a hidden Linux VM. The plugin shares the VM's host networking, so it reaches the MinIO container through its **published host port** at `host.docker.internal:9000` (not the compose service name):

```
┌──────────────── Docker Desktop VM ─────────────────┐
│ Docker Engine (daemon)                              │
│  ├─ rclone plugin  ── remote "minio" (provider=Minio)│
│  │        │  S3 over http://host.docker.internal:9000 │
│  ├─ minio container  ── -p 9000:9000 (S3 API)        │
│  │        └─ bucket "agent-storage"                  │
│  └─ sandbox container                                │
│        mounts volume → /workspace/storage  ──────────┘
│        (writes become objects in the bucket)
└─────────────────────────────────────────────────────┘
```

> A bind mount can't reach MinIO here: `gcs()`-style bind volumes resolve their `hostPath` on the daemon host, which on Docker Desktop is the hidden VM. The rclone **volume plugin** does the S3 I/O *inside* that VM, so it's the one filesystem-style path that works on macOS/Windows. On a Linux daemon host both options work.

## 1. Run MinIO locally and create the bucket

`docker-compose.minio.yml`:

```yaml
services:
  minio:
    image: minio/minio
    command: server /data --console-address ":9001"
    environment:
      MINIO_ROOT_USER: minioadmin
      MINIO_ROOT_PASSWORD: minioadmin
    ports:
      - "9000:9000" # S3 API — published so the rclone plugin can reach it
      - "9001:9001" # web console at http://localhost:9001
    volumes:
      - minio-data:/data
  createbucket:
    image: minio/mc
    depends_on: [minio]
    entrypoint: >
      /bin/sh -c "
      until mc alias set local http://minio:9000 minioadmin minioadmin; do sleep 1; done;
      mc mb -p local/agent-storage;
      exit 0;
      "
volumes:
  minio-data:
```

```bash
docker compose -f docker-compose.minio.yml up -d
```

The `createbucket` one-shot waits for MinIO, creates `agent-storage`, and exits. MinIO's S3 API is now published on the host at `localhost:9000`; the console is at `localhost:9001`.

## 2. Install the rclone volume plugin

Use the tag matching your daemon's architecture (`arm64` on Apple Silicon, `amd64` on Intel/most Linux):

```bash
docker plugin install rclone/docker-volume-rclone:arm64 \
  args="-v" --alias rclone --grant-all-permissions
```

## 3. Configure the `[minio]` remote, then enable

The plugin reads its config from `rclone.conf` inside its config mount. **Write the config before enabling** — see the gotcha below.

`provider = Minio` selects path-style addressing (required for MinIO). For local dev the endpoint is the published host port as seen from inside the VM:

```ini
[minio]
type = s3
provider = Minio
access_key_id = minioadmin
secret_access_key = minioadmin
endpoint = http://host.docker.internal:9000
```

**On a Linux daemon host**, that file lives on the host — write it directly:

```bash
sudo mkdir -p /var/lib/docker-plugins/rclone/config
sudo tee /var/lib/docker-plugins/rclone/config/rclone.conf >/dev/null <<'EOF'
[minio]
type = s3
provider = Minio
access_key_id = minioadmin
secret_access_key = minioadmin
endpoint = http://host.docker.internal:9000
EOF
```

**On Docker Desktop (macOS/Windows)**, that path is inside the VM, not on your machine. Write it through a one-shot privileged helper that enters the VM's namespace:

```bash
docker run --rm --privileged --pid=host justincormack/nsenter1 /bin/sh -c '
cat > /var/lib/docker-plugins/rclone/config/rclone.conf <<EOF
[minio]
type = s3
provider = Minio
access_key_id = minioadmin
secret_access_key = minioadmin
endpoint = http://host.docker.internal:9000
EOF'
```

Then enable (or, if the plugin was already enabled, `disable` then `enable` to reload):

```bash
docker plugin enable rclone
```

> **Gotcha — a bad remote stops the plugin from starting at all.** `rclone serve docker` parses the *entire* config on startup; a stale or unreachable remote (e.g. an expired OAuth token left over from another backend) can make it hang before it ever creates its socket, so **every** `docker plugin enable` fails with `dial unix …/rclone.sock: connect: no such file or directory` — even for an unrelated volume. If you hit this, inspect `/var/lib/docker-plugins/rclone/config/rclone.conf` and remove the broken stanza. A managed plugin only re-reads its config on `disable`/`enable`.

## 4. Attach the volume to a sandbox

For local development, let the sandbox create and clean up the volume with `lifecycle: 'managed'`. The `driver`/`driverOptions` map straight onto `docker volume create --driver rclone --opt …`:

```typescript
import { createDockerSandbox } from '@deepagents/context';

await using sandbox = await createDockerSandbox({
  image: 'alpine:latest',
  volumes: [
    {
      type: 'volume',
      name: 'minio-agent-storage',
      containerPath: '/workspace/storage',
      readOnly: false,
      lifecycle: 'managed', // sandbox runs `docker volume create` and removes it on dispose
      driver: 'rclone',
      driverOptions: {
        remote: 'minio:agent-storage', // "<remote name>:<bucket>"
        'vfs-cache-mode': 'writes',
      },
    },
  ],
});

await sandbox.executeCommand(
  'echo "result" > /workspace/storage/output.txt',
);
```

`remote: 'minio:agent-storage'` is two parts: `minio` is the remote from your `rclone.conf`; `agent-storage` is the bucket. The S3 credentials and endpoint stay in the plugin's config — never in your TypeScript. The file written at `/workspace/storage/output.txt` becomes the object `agent-storage/output.txt` in MinIO (visible in the console at `localhost:9001`).

> Pass the **named remote** via `remote=`, not an inline backend config. The plugin rejects `-o type=s3 -o provider=…` (`unsupported backend option`), and an on-the-fly `:s3:` remote with separate `-o s3-*` options silently falls back to AWS instead of your endpoint. Define the remote in `rclone.conf` and reference it by name.

### Alternative: a pre-created (`external`) volume

To share one volume across runs or pre-provision it, create it yourself and reference it with `lifecycle: 'external'` (the form the GCS recipe uses):

```bash
docker volume create minio-agent-storage -d rclone \
  -o remote=minio:agent-storage \
  -o vfs-cache-mode=writes
```

```typescript
volumes: [
  {
    type: 'volume',
    name: 'minio-agent-storage',
    containerPath: '/workspace/storage',
    readOnly: false,
    lifecycle: 'external', // already exists; DeepAgents only inspects it
  },
],
```

The bucket-to-volume mapping lives entirely in how you create the volume, so per-agent isolation (a bucket each, or a prefix each via `remote=minio:shared/agent-a`) needs no DeepAgents change.

## Real or remote MinIO

For a MinIO server that isn't a local container, only the `rclone.conf` stanza changes — swap the endpoint and credentials, and use HTTPS in production:

```ini
[minio]
type = s3
provider = Minio
access_key_id = YOUR_ACCESS_KEY
secret_access_key = YOUR_SECRET_KEY
endpoint = https://minio.internal.example.com
```

If the daemon runs on a remote host (e.g. a VM you target with `DOCKER_CONTEXT`), use a named `external` volume — a bind mount's `hostPath` would be validated on the app server but resolved on the VM. See the [GCS recipe's remote-daemon section](/docs/context/recipes/gcs-cloud-storage#remote-daemon-on-a-gcp-vm) for the docker-context setup; everything there applies, with the `[minio]` stanza in place of `[gcs]`.

## Things to know

- **Writes are eventually consistent.** With `vfs-cache-mode=writes`, a write lands in the plugin's local cache and uploads asynchronously (after `--vfs-write-back`, default `5s`, plus a flush on unmount). Don't assert an object exists the instant a file closes; a long-lived sandbox flushes within a few seconds, while a container that exits immediately may upload a moment later. Lower `--vfs-write-back` for snappier uploads.
- **This is FUSE over object storage.** Whole-file create-and-close writes are the reliable pattern; tools doing in-place random writes can misbehave.
- **`provider = Minio` is what makes it work.** It switches rclone to path-style addressing; without it, requests target `bucket.host…` virtual-host style and fail against MinIO.
- **No MinIO coupling in the library.** `@deepagents/context` only attaches an ordinary Docker named volume. Switching backend (S3, R2, Wasabi, Azure) is a one-stanza `rclone.conf` change — `type`, `provider`, `endpoint` — with no code change.
