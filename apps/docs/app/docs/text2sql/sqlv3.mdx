title: Caller-Owned Sandbox
description: Compose the SQL sandbox extension with your own commands, plugins, and backends
---

`Text2Sql` uses a caller-owned sandbox. `@deepagents/text2sql` exposes a
composable `sqlSandboxExtension(adapter)` — a bundle of the `sql` subcommand,
its transform plugins, and its arg-repair hook. Compose it into any backend
via `createBashTool` + `createRoutingSandbox` with `hostExtensions`.

## What This Path Gives You

- Full control over sandbox construction (`fs`, `cwd`, `env`, and custom commands)
- Manual registration of `createSqlCommand(adapter)`
- Explicit plugin control (`SqlBacktickRewritePlugin`, `SqlProxyEnforcementPlugin`)
- Swappable backends — virtual (just-bash), Docker, Agent OS, or any `Sandbox` implementation

## Quick Start

```typescript
import { groq } from '@ai-sdk/groq';
import { InMemoryFs } from 'just-bash';

import {
  ContextEngine,
  InMemoryContextStore,
  createBashTool,
  createRoutingSandbox,
  createVirtualSandbox,
} from '@deepagents/context';
import { Text2Sql, sqlSandboxExtension } from '@deepagents/text2sql';

const store = new InMemoryContextStore();
const model = groq('openai/gpt-oss-20b');

const sandbox = await createBashTool({
  sandbox: await createRoutingSandbox({
    backend: await createVirtualSandbox({ fs: new InMemoryFs() }),
    hostExtensions: [sqlSandboxExtension(adapter)],
  }),
});

const text2sql = new Text2Sql({
  version: 'v3-demo',
  adapter,
  sandbox,
  model,
  context: (...fragments) => {
    const engine = new ContextEngine({
      store,
      chatId: 'chat-123',
      userId: 'user-456',
    });
    engine.set(...fragments);
    return engine;
  },
});
```

## Composing With Your Own Extensions

`mergeExtensions` concatenates `commands` and `plugins`, chains
`onBeforeBashCall` hooks (each sees the prior hook's output), and merges `env`
with last-wins semantics. Your extensions sit alongside SQL's.

```typescript
import type { SandboxExtension } from '@deepagents/context';

const myExtension: SandboxExtension = {
  commands: [myCustomCommand],
  plugins: [myAuditPlugin],
  onBeforeBashCall: ({ command }) => ({ command: myRewrite(command) }),
  env: { MY_FLAG: '1' },
};

const sandbox = await createBashTool({
  sandbox: await createRoutingSandbox({
    backend: await createVirtualSandbox({ fs: new InMemoryFs() }),
    hostExtensions: [sqlSandboxExtension(adapter), myExtension],
  }),
});
```

## Swapping the Backend (Docker / Agent OS)

The sandbox composition shape is the same for every backend — only the
`backend` argument to `createRoutingSandbox` changes. Virtual (just-bash),
Docker, Agent OS, and any other implementation of `bash-tool`'s `Sandbox`
interface all plug in the same way.

```typescript
import {
  createBashTool,
  createDockerSandbox,
  createRoutingSandbox,
} from '@deepagents/context';
import { sqlSandboxExtension } from '@deepagents/text2sql';

const sandbox = await createBashTool({
  sandbox: await createRoutingSandbox({
    backend: await createDockerSandbox({ image: 'alpine' }),
    hostExtensions: [sqlSandboxExtension(adapter)],
  }),
});
```

Host commands (like `sql validate`, `sql run`) execute in Node — the backend is
only used for file I/O and pass-through commands. File events are only
observable on the virtual backend; wrap its `fs` in `ObservedFs` when you need
them (see [File Event Metadata](#file-event-metadata)).

### Pipeline Boundary

On Docker and Agent OS backends, host commands dispatch only at the top of a
line. `sql run "..." | jq .` works on the virtual backend (bash parses the
pipeline and matches `sql` inside) but on Docker/Agent OS the entire line runs
in the container, which has no `sql` binary. Pipe in two turns
(`sql run ...` then `cat /sql/<uuid>.json | jq .`), or use the virtual backend.

## Low-Level Primitives

If you need something `sqlSandboxExtension` does not model, the underlying
pieces are all exported: `createSqlCommand(adapter)`,
`SqlBacktickRewritePlugin`, `SqlProxyEnforcementPlugin`. Compose them directly
into any sandbox backend that accepts custom commands, plugins, and a
pre-call hook.

## File Event Metadata

Wrap your filesystem with `ObservedFs` and expose `drainFileEvents` on the
returned sandbox. During `chat()`, the context chat pipeline drains those file
events and stores them on the assistant message as `metadata.fileEvents`.

```typescript
import {
  ObservedFs,
  createBashTool,
  createRoutingSandbox,
  createVirtualSandbox,
} from '@deepagents/context';
import { InMemoryFs } from 'just-bash';

const observed = new ObservedFs(new InMemoryFs());
const base = await createBashTool({
  sandbox: await createRoutingSandbox({
    backend: await createVirtualSandbox({ fs: observed }),
    hostExtensions: [sqlSandboxExtension(adapter)],
  }),
});
const sandbox = { ...base, drainFileEvents: () => observed.drain() };
```

```typescript
interface FileEvent {
  path: string;
  op: 'read' | 'write' | 'delete' | 'modify';
  timestamp: number;
}
```

When no file operations occur in a turn, `fileEvents` is omitted.

## Current Scope

- File event tracking is currently scoped to just-bash-backed sandboxes.
- The SQL interface remains a bash subcommand (`sql validate`, `sql run`), not
  a standalone AI SDK tool.
- Multi-adapter routing is still out of scope.
