---
title: Text2SqlV3 (Experimental)
description: Use a caller-owned sandbox and wire the sql command yourself
---

`Text2SqlV3` is an experimental alternative to `Text2Sql` for apps that already
own a bash sandbox. Instead of letting `createResultTools` assemble an internal
just-bash instance plus filesystem, you wire the `sql` subcommand onto your own
`Bash` instance and pass the resulting `AgentSandbox` straight into
`Text2SqlV3`.

Use `Text2Sql` when you want the package to manage the filesystem-backed result
tools for you. Use `Text2SqlV3` when you already have a shared sandbox, custom
commands, or extra bash policies that Text2SQL should reuse unchanged.

## What Changes vs `Text2Sql`

- You own the `Bash` instance and its filesystem end-to-end.
- You register `createSqlCommand(adapter)` plus the SQL transform plugins yourself.
- `Text2SqlV3.chat()` passes your sandbox directly into `agent()` with no extra wrapper layer.
- Assistant `metadata.createdFiles` comes from hidden bash-tool metadata (`resultPath`) instead of `TrackedFs`.

## Quick Start

```typescript
import { groq } from '@ai-sdk/groq';
import { Bash, InMemoryFs } from 'just-bash';

import {
  ContextEngine,
  InMemoryContextStore,
  createBashTool,
} from '@deepagents/context';
import {
  Text2SqlV3,
  createSqlCommand,
  SqlBacktickRewritePlugin,
  SqlProxyEnforcementPlugin,
} from '@deepagents/text2sql';

const store = new InMemoryContextStore();
const model = groq('openai/gpt-oss-20b');
const { command: sqlCommand, repair: repairSql } = createSqlCommand(adapter);

const bash = new Bash({
  fs: new InMemoryFs(),
  customCommands: [sqlCommand],
});

bash.registerTransformPlugin(new SqlBacktickRewritePlugin());
bash.registerTransformPlugin(new SqlProxyEnforcementPlugin());

const sandbox = await createBashTool({
  sandbox: bash,
  onBeforeBashCall: ({ command }) => ({ command: repairSql(command) }),
});

const text2sql = new Text2SqlV3({
  version: 'v3-demo',
  adapter,
  sandbox,
  model,
  context: (...fragments) => {
    const engine = new ContextEngine({
      store,
      chatId: 'chat-123',
      userId: 'user-456',
    });
    engine.set(...fragments);
    return engine;
  },
});
```

## Config Differences

`Text2SqlV3Config` keeps the familiar `adapter`, `context`, `version`, `model`,
and optional `tools` / `transform` fields. The main change is that v3 requires a
ready-made `sandbox: AgentSandbox` and removes the old `filesystem` option.

| Field | Status in v3 |
| --- | --- |
| `adapter` | Same as `Text2Sql` |
| `sandbox: AgentSandbox` | Required |
| `context` | Same as `Text2Sql` |
| `version` | Same as `Text2Sql` |
| `tools?: RenderingTools` | Same as `Text2Sql` |
| `model` | Same as `Text2Sql` |
| `transform?` | Same as `Text2Sql` |
| `filesystem` | Removed |

## How `createdFiles` Works

`Text2Sql` v1 captures created files through `TrackedFs`. `Text2SqlV3` has no
internal filesystem layer, so the `sql run` handler records
`useBashMeta()?.setHidden({ resultPath })` on each successful execution.
`Text2SqlV3.chat()` reads those hidden `resultPath` values from
`result.steps[].toolResults[].output.meta` and copies them into the final
assistant message metadata.

That keeps the assistant message shape stable while letting the caller keep full
control over the sandbox implementation.

## Current Scope

- `sql` remains a bash subcommand, not a first-class AI SDK tool.
- Multi-adapter routing is still out of scope for v3.
- `Text2Sql` v1 remains supported; v3 is an opt-in alternative.
