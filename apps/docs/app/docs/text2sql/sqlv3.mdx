---
title: SQL CLI in Sandboxes
description: Run Text2SQL chat flows with the real sql command installed inside the sandbox
---

Text2SQL chat flows use a caller-owned sandbox. The `sql` command is a real
CLI from `@deepagents/text2sql`; it is not a host-routed sandbox extension.
Install the package in the sandbox before the agent tries to run:

```bash
sql validate <db> "SELECT ..."
sql run <db> "SELECT ..."
```

Adapter-map keys become the `<db>` selector. They must match
`/^[A-Za-z_][A-Za-z0-9_]*$/`; if you build the map dynamically, validate the
keys with `isValidAdapterName(name)` or `validateAdapterNames(names)`.

## Container Setup

```typescript
import {
  ContextEngine,
  InMemoryContextStore,
  createContainerTool,
  npm,
} from '@deepagents/context';
import { createSqlCommandHooks } from '@deepagents/text2sql';

const sandbox = await createContainerTool({
  installers: [npm('@deepagents/text2sql', { ensureRuntime: true })],
  env: {
    TEXT2SQL_ADAPTERS: '/workspace/text2sql-adapters.ts',
  },
  ...createSqlCommandHooks({ adapters: { main: adapter } }),
});

const context = new ContextEngine({
  store: new InMemoryContextStore(),
  chatId: 'chat-123',
  userId: 'user-456',
});
```

`TEXT2SQL_ADAPTERS` must point to a module inside the sandbox whose default
export is `Record<string, Adapter>`. Mount or upload that file alongside any
database files or credentials the adapter needs. If you already mount your app
source into the container, point the env var at that mounted module path.

The host-side `createSqlCommandHooks()` call repairs common model quoting
mistakes before the command reaches bash, blocks raw database CLIs, and attaches
formatted-SQL metadata to `sql validate` and `sql run` results. It should use
the same adapter map that the sandbox adapter module exports.

## CLI Commands

```bash
sql index [adapter ...]
sql validate <db> "SELECT ..."
sql run <db> "SELECT ..." [--out-dir ./sql]
```

`sql index` writes a JSON manifest to stdout. The manifest includes
`fragmentsPath`, `eventsPath`, `adapters`, and the fragment count. Read
`fragmentsPath` and pass those fragments to your `ContextEngine` with
`instructions()` before the first `chat()` call.

## File Event Metadata

File event tracking is scoped to virtual sandboxes backed by `ObservedFs`.
Commands installed inside Docker or Agent OS do real filesystem work in those
environments, but they do not emit `ObservedFs` events.

```typescript
import { InMemoryFs } from 'just-bash';

import {
  ObservedFs,
  createBashTool,
  createVirtualSandbox,
} from '@deepagents/context';

const observed = new ObservedFs(new InMemoryFs());
const base = await createBashTool({
  sandbox: await createVirtualSandbox({ fs: observed }),
});
const sandbox = { ...base, drainFileEvents: () => observed.drain() };
```

When no file operations occur in a turn, `fileEvents` is omitted.

## Current Scope

- The SQL interface is a real bash command installed from the package.
- Missing `sql` means the sandbox was not prepared correctly; install
  `@deepagents/text2sql` before starting the agent.
- OpenAPI-style command surfaces should also be generated or installed as real
  CLIs, not host-routed through bash.
