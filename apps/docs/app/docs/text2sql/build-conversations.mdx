The `chat()` method streams answers and persists each turn through the
`ContextEngine` you pass to `Text2Sql`. Thread identity comes from the
`chatId` and `userId` on that engine, not from the `Text2Sql` instance.
Use `chat()` when you want a durable thread. `toSql()` stays stateless and only
uses the fragments you pass into that one call. Before each `chat()` call,
append only the new incoming message for that turn with `engine.continue(...)`.

## Basic Usage

```typescript
await engine.continue(user('Show me top 10 customers'));
const stream = await text2sql.chat({ generateTitle: true });

for await (const chunk of stream) {
  if (chunk.type === 'text-delta') {
    process.stdout.write(chunk.textDelta);
  }
}
```

That `engine.continue(...)` call should contain only the current turn payload,
not the full thread transcript.

## Indexing Progress

`chat()` indexes schema context before the assistant response starts. That work
can be expensive on large databases, so Text2SQL emits typed progress chunks in
the same UI stream:

```typescript
import {
  TEXT2SQL_INDEX_PROGRESS_CHUNK,
  type Text2SqlIndexProgressEvent,
} from '@deepagents/text2sql';

await engine.continue(user('Show me top 10 customers'));
const stream = await text2sql.chat();

for await (const chunk of stream) {
  if (chunk.type === TEXT2SQL_INDEX_PROGRESS_CHUNK) {
    const event = chunk.data as Text2SqlIndexProgressEvent;

    renderIndexingStatus({
      type: event.type,
      adapter: event.adapter,
      phase: event.phase,
      table: event.table,
      message: event.message,
      current: event.current,
      total: event.total,
      cached: event.cached,
      timestampMs: event.timestampMs,
    });

    continue;
  }

  renderAssistantChunk(chunk);
}
```

Progress events include `index:start`, `index:end`, `adapter:start`,
`adapter:end`, `adapter:cache-hit`, `adapter:cache-miss`, `phase:start`,
`phase:progress`, `phase:end`, `adapter:error`, and `index:error`.

Use the Unix epoch `timestampMs` to derive durations in the client. For example,
subtract the `timestampMs` on a `phase:start` event from the matching
`phase:end` event for that adapter and phase. Text2SQL does not emit
`elapsedMs`, because UIs may want to group timing by index, adapter, phase,
table, or their own display model.

## Reusing the Same Thread

Reuse the same store plus the same `chatId` and `userId` whenever you want a
follow-up request to see prior messages:

```typescript
import { groq } from '@ai-sdk/groq';
import { InMemoryFs } from 'just-bash';

import {
  ContextEngine,
  SqliteContextStore,
  createBashTool,
  createRoutingSandbox,
  createVirtualSandbox,
  user,
} from '@deepagents/context';
import { Text2Sql, sqlSandboxExtension } from '@deepagents/text2sql';

const store = new SqliteContextStore('./text2sql-chat.sqlite');

async function createThread(chatId: string, userId: string) {
  // Reuse your configured database adapter from Getting Started.
  const sandbox = await createBashTool({
    sandbox: await createRoutingSandbox({
      backend: await createVirtualSandbox({ fs: new InMemoryFs() }),
      hostExtensions: [sqlSandboxExtension({ main: adapter })],
    }),
  });

  const engine = new ContextEngine({
    store,
    chatId,
    userId,
  });
  const text2sql = new Text2Sql({
    version: 'v1',
    model: groq('openai/gpt-oss-20b'),
    sandbox,
    adapters: { main: adapter },
    context: engine,
  });

  return { engine, text2sql };
}

const firstTurn = await createThread('chat-123', 'user-456');
await firstTurn.engine.continue(user('Show me total revenue'));
for await (const _ of await firstTurn.text2sql.chat()) {
  // drain or stream chunks to your UI
}

const followUp = await createThread('chat-123', 'user-456');
await followUp.engine.continue(user('Break that down by quarter'));
for await (const _ of await followUp.text2sql.chat()) {
  // drain or stream chunks to your UI
}
```

## How It Works

When you call `chat()`:

1. **Load stored context** - The `ContextEngine` resolves prior messages for the same thread.
2. **Inject SQL guidance** - Text2SQL adds its built-in workflow guidance plus cached schema fragments.
3. **Run tools** - The agent can validate SQL, execute read-only queries, and inspect saved results.
4. **Persist the turn** - `engine.continue(...)` saves the incoming message and reserves the assistant slot before streaming starts, then the assistant reply plus chat metadata are finalized when the turn finishes.

## Message Format

`ContextEngine.continue()` accepts `ChatMessage` from `@deepagents/context`. In
most apps, `user('...')` or `user(uiMessage)` is enough. When prior turns are
already stored in your `ContextEngine`, do not resend them:

```typescript
await engine.continue(user('Show me sales by region'));
const stream = await text2sql.chat();
```

If your UI keeps a full transcript client-side, strip it down to the new
message for this request before calling `engine.continue(...)`.

## Options

`chat()` accepts an optional options object:

| Option | Type | Description |
| ------ | ---- | ----------- |
| `abortSignal` | `AbortSignal` | Cancels title generation, repair attempts, and the active stream |
| `generateTitle` | `boolean` | Uses the model to generate the first chat title instead of the static fallback |

### Abort Signal

```typescript
const controller = new AbortController();

await engine.continue(user('Show me top 10 customers'));
const stream = await text2sql.chat({ abortSignal: controller.signal });

setTimeout(() => controller.abort(), 10_000);
```

The same signal is forwarded to title generation and tool-call repair, so cancellation stops the full turn rather than only the visible stream.

## Error Handling

Tool and validation errors are surfaced as user-facing stream messages. After a
cancellation or failure, start a new `chat()` call. Reuse the same `chatId` to
continue the thread, or use a new `chatId` to start a separate conversation.

## Integration Example

```typescript
app.post('/api/chat', async (req, res) => {
  const { message, chatId } = req.body;
  const userId = req.user.id;

  const { engine, text2sql } = await createThread(chatId, userId);
  await engine.continue(user(message));
  const stream = await text2sql.chat({ generateTitle: true });

  res.setHeader('Content-Type', 'text/event-stream');

  for await (const chunk of stream) {
    res.write(`data: ${JSON.stringify(chunk)}\n\n`);
  }

  res.end();
});
```

`message` should be only the new turn payload for this request.

## Best Practices

1. **Keep `chatId` stable per conversation** so follow-up turns can resolve the same history.
2. **Keep `userId` stable per signed-in user** so related chats remain attributable.
3. **Share one store across requests** if you want persistence between processes or server restarts.
4. **Create a fresh `chatId` for a new topic** instead of overloading an existing thread.
