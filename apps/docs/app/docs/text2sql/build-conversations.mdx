The `chat()` method streams answers and persists each turn through the
`ContextEngine` returned by your `context` factory. Thread identity comes from
the `chatId` and `userId` on that engine, not from the `Text2Sql` instance.
Use `chat()` when you want a durable thread. `toSql()` stays stateless and only
uses the fragments you pass into that one call.

## Basic Usage

```typescript
const stream = await text2sql.chat(
  [{ role: 'user', content: 'Show me top 10 customers' }],
  { generateTitle: true },
);

for await (const chunk of stream) {
  if (chunk.type === 'text-delta') {
    process.stdout.write(chunk.textDelta);
  }
}
```

## Reusing the Same Thread

Reuse the same store plus the same `chatId` and `userId` whenever you want a
follow-up request to see prior messages:

```typescript
import { groq } from '@ai-sdk/groq';
import { InMemoryFs } from 'just-bash';

import { ContextEngine, SqliteContextStore } from '@deepagents/context';
import { Text2Sql } from '@deepagents/text2sql';

const store = new SqliteContextStore('./text2sql-chat.sqlite');

function createText2Sql(chatId: string, userId: string) {
  return new Text2Sql({
    version: 'v1',
    model: groq('openai/gpt-oss-20b'),
    filesystem: new InMemoryFs(),
    // Reuse your configured database adapter from Getting Started.
    adapter,
    context: (...fragments) => {
      const engine = new ContextEngine({
        store,
        chatId,
        userId,
      });
      engine.set(...fragments);
      return engine;
    },
  });
}

const firstTurn = createText2Sql('chat-123', 'user-456');
await firstTurn.chat([{ role: 'user', content: 'Show me total revenue' }]);

const followUp = createText2Sql('chat-123', 'user-456');
await followUp.chat([
  { role: 'user', content: 'Break that down by quarter' },
]);
```

## How It Works

When you call `chat()`:

1. **Load stored context** - The `ContextEngine` resolves prior messages for the same thread.
2. **Inject SQL guidance** - Text2SQL adds its built-in workflow guidance plus cached schema fragments.
3. **Run tools** - The agent can validate SQL, execute read-only queries, and inspect saved results.
4. **Persist the turn** - The incoming message is saved before streaming starts, then the assistant reply plus chat metadata are finalized when the turn finishes.

## Message Format

`chat()` accepts `ChatMessage[]` from `@deepagents/context`. In most apps, plain
`{ role, content }` objects are enough:

```typescript
await text2sql.chat([
  { role: 'user', content: 'Show me sales by region' },
  { role: 'assistant', content: 'California leads the list.' },
  { role: 'user', content: 'Now show me just California by month' },
]);
```

## Options

`chat()` accepts an optional second argument:

| Option | Type | Description |
| ------ | ---- | ----------- |
| `abortSignal` | `AbortSignal` | Cancels title generation, repair attempts, and the active stream |
| `generateTitle` | `boolean` | Uses the model to generate the first chat title instead of the static fallback |

### Abort Signal

```typescript
const controller = new AbortController();

const stream = await text2sql.chat(messages, {
  abortSignal: controller.signal,
});

setTimeout(() => controller.abort(), 10_000);
```

The same signal is forwarded to title generation and tool-call repair, so cancellation stops the full turn rather than only the visible stream.

## Error Handling

Tool and validation errors are surfaced as user-facing stream messages. After a
cancellation or failure, start a new `chat()` call. Reuse the same `chatId` to
continue the thread, or use a new `chatId` to start a separate conversation.

## Integration Example

```typescript
app.post('/api/chat', async (req, res) => {
  const { messages, chatId } = req.body;
  const userId = req.user.id;

  const text2sql = createText2Sql(chatId, userId);
  const stream = await text2sql.chat(messages, { generateTitle: true });

  res.setHeader('Content-Type', 'text/event-stream');

  for await (const chunk of stream) {
    res.write(`data: ${JSON.stringify(chunk)}\n\n`);
  }

  res.end();
});
```

## Best Practices

1. **Keep `chatId` stable per conversation** so follow-up turns can resolve the same history.
2. **Keep `userId` stable per signed-in user** so related chats remain attributable.
3. **Share one store across requests** if you want persistence between processes or server restarts.
4. **Create a fresh `chatId` for a new topic** instead of overloading an existing thread.
